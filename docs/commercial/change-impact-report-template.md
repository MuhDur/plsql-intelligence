# Change Impact Assessment Report — Template

**PLSQL-CVT-002** — canonical Markdown source for the Change Impact
Assessment artefact the engagement (PLSQL-CVT-001 §1.2) delivers.
The template is rendered against the synthetic lab corpus L1 + L2
to validate that every required field has a producer.

This file is the **shape spec**; the production report is generated
by `plsql lineage report --output change-impact.md --against
<changeset>`. Reviewers diff the Markdown — the HTML/PDF forms are
derived.

## 1. Executive Summary

One paragraph (≤ 6 sentences) generated by `plsql cicd predict
--summarise`. Required content:

* The overall risk verdict.
* The count of changed objects by category.
* The count of predicted invalidations by confidence tier.
* Any `UnknownReason` discriminators that fired (R13).

The summary must end with the **operator decision prompt**: a
single sentence stating whether the report recommends `proceed`,
`gate-review`, or `block`.

## 2. Trust Block

Per plan §1.5 — the trust block lists the analysis's known
limitations up front so the reader can calibrate every later
claim against it.

| Completeness counter | Value |
|---|---|
| `objects_parsed` | `<n>` |
| `objects_skipped_due_to_wrap` | `<n>` |
| `objects_skipped_due_to_parser_recover` | `<n>` |
| `dynamic_sql_sites` | `<n>` |
| `opaque_dynamic_sql_sites` | `<n>` |
| `db_link_edges` | `<n>` |
| `unresolved_references` | `<n>` |
| `catalog_available` | `true` / `false` |
| `plscope_available` | `true` / `false` |

Followed by the typed `UnknownReason` discriminator counts. Any
non-zero count routes the reader to §6.

## 3. Changeset

Generated from the `ChangeSet` envelope (PLSQL-CICD-001 + the
`parse_change_file` parser, PLSQL-LIN-007).

For each changed object:

* Owner.Name (logical id).
* `ChangedObjectKind` discriminator.
* `previous_hash` / `new_hash` (sha256).
* Source file path (project-relative).

Sort: stable by `(owner, name)` so re-runs diff cleanly.

## 4. Predicted Invalidations

Generated from `plsql cicd predict --robot-json`. Each row carries
its origin so the reader can trace it back:

| Owner.Object | Type | Distance | Confidence | Reason | Action |
|---|---|---|---|---|---|
| `<schema>.<name>` | `<OBJECT_TYPE>` | `<n>` hops | `High`/`Medium`/`Low` | `<InvalidationReason>` | `ALTER … COMPILE` / `lazy recompile` |

Sort order: `(distance ascending, confidence descending, owner,
name)`.

## 5. Deployment Plan

Topologically-sorted DDL emitted by `plsql cicd plan`
(PLSQL-CICD-003). The plan section is the operator-facing
companion to the script the engine generates; both must use the
same ordinals so the audit log lines up.

```sql
-- 1. <kind weight 0: sequences>
-- 2. <kind weight 1: types>
-- 3. <kind weight 2: tables>
-- ...
-- (full table per `plsql cicd plan --markdown`)
```

Notes (from `DeploymentPlan.notes`) appear underneath as a bullet
list — typically:

* `<N>` unclassified file(s) present — review before deploy.
* `<N>` uncertainty record(s) (R13) — gate before deploy.

## 6. Uncertainties (R13)

Every `UnknownReason` discriminator that fired during prediction
gets its own subsection. Each subsection MUST include:

* The variant tag (`DynamicSqlOpaque`, `DbLinkRemoteObject`,
  `WrappedSource`, `MissingCatalogObject`, …).
* A short explanation of what the engine cannot see.
* The affected object(s) by logical id.
* A recommended next step (e.g. "supply PL/Scope dump",
  "include the package body in the next checkout").

This section is the contract the engine offers to the reader:
**the analysis is honest about what it does not know.**

## 7. Recompile Order

The output of `plsql lineage recompile_order` — a topologically
sorted list of the objects that must be recompiled after the
changeset is applied, plus the cycles the engine couldn't order
and the inputs it didn't find in the graph.

| Position | Owner.Object | Force compile? |
|---|---|---|
| 1 | `<schema>.<name>` | `true` / `false` |

Cycles (if any) and missing inputs are rendered in dedicated
sub-tables underneath.

## 8. Call Graph & Table Usage

For each changed PL/SQL unit, inline an SVG produced by
`render_call_graph_svg` (PLSQL-DOC-006). For each changed table,
inline an SVG from `render_table_usage_svg` (PLSQL-DOC-007). The
inline SVG renderers ship deterministic, stylesheet-free output
so the embedded images render identically across viewers.

## 9. Lineage Verdict per Hot Question

Three hot questions every Change Impact Assessment must answer:

1. **What breaks if I change this object?** — table from
   `plsql_lineage::impact`. Confidence column required.
2. **What does this object depend on?** — table from
   `plsql_lineage::dependencies`.
3. **What columns are read and written?** — table from
   `plsql_lineage::column_readers` and `column_writers`,
   pre-formatted with the per-column resolution-error column
   (`ColumnAccessResult::resolution_error`).

## 10. AUDIT statement (operator-actionable)

The report MUST end with a synthesised `AUDIT` block the operator
can paste into a deployment ticket. The block carries:

* The exact engine version + SHA (from the cover).
* The overall risk verdict (from §1).
* The unresolved / uncertainty counts (from §2).
* The recompile order positions (from §7).
* A SHA-256 of the JSON envelope this report was rendered from
  (`plsql.cicd.deployment_plan` / `plsql.cicd.invalidation_prediction`
  / `plsql.lineage.impact`).

The AUDIT block is the artefact the customer attaches to their
production change ticket; it's the single artifact a reviewer
needs to recover provenance.

## Appendix A — Rendering against the synthetic lab

The template is exercised against the **L1** + **L2** corpora to
prove every section has a producer:

* L1 (PLSQL-LAB-001) — 10 packages + 5 views + 2 triggers. The
  PLSQL-LAB-002 hero diff (rename `p_emp_id` →
  `p_employee_id` on `employee_mgmt.fire_employee` /
  `get_salary`) drives Sections 3–10.
* L2 (PLSQL-LAB-004) — synonyms, ACCESSIBLE BY, rights mix,
  `DBMS_ASSERT` scenarios, `EXECUTE IMMEDIATE` with/without
  binds. Section 2's Trust Block and Section 6's
  `UnknownReason::DynamicSqlOpaque` records get populated against
  this corpus.

CI gate: `tools/plan-lint` enforces structural conformance — any
section heading that drifts from the table above blocks PR merge.

## Appendix B — Variants

The report ships in three forms produced from the same JSON
envelope:

* **Markdown** (this template).
* **HTML** — same shape via `render_object_html` family.
* **JSON envelope** — `plsql.cicd.change_impact_report` v1
  (defined in the `plsql-output` schema-version registry).

Customers pick the form their ticket system accepts.
